一个陪伴型 Agent 和用户连续聊了三天，累计上下文早就超过 100k token。第三天用户突然问："我周二提到的那个想去的地方，具体在哪来着？"我把这句话当成对记忆系统最好的压力测试：简单截断的话，周二的对话早被挤出窗口了；全量 Summary 的话，"那个地方"的具体地名多半在压缩中被抹成了"聊了旅行计划"。要同时答对，记忆系统必须能按**时间**定位到周二，再按**语义**从那天的内容里捞出地名。这篇文章讲的就是支撑这种查询的分层架构和它背后的写入、检索、遗忘算法。

## 为什么截断和全量摘要都不够

朴素方案有两类，各丢一半信息。滑动窗口截断保留最近 N 个 token，特点是细节完整但记不住远处——周二的对话直接不在窗口里。全量摘要把历史压成一段话，特点是覆盖长但精度低——地名、金额、时间这类"限定词"最容易在摘要里被吞掉。真正的问题在于：这两种做法都用**单一时间维度**管理信息，而用户的查询往往是"某个时间点 + 某个主题"的联合检索。解法不是在截断和摘要之间调参数，而是把记忆拆成生命周期不同的几层，让每层各自负责它擅长的召回方式。

## 三层记忆的定义与生命周期

我把记忆分成三层，边界由**存活时长**和**存储形态**共同定义：

| 层 | 存的是什么 | 形态 | 生命周期 | 召回方式 |
|---|---|---|---|---|
| Working Memory | 当前窗口内的原始轮次 | 明文 token | 会话内，随窗口滑出即逝 | 直接在 prompt 里 |
| Short-term Memory | 近期会话的结构化摘要 | 摘要 + 元数据 | 数天到数周，滚动过期 | 按会话/日期取 |
| Long-term Memory | 已验证的稳定事实 | 向量 + 结构化字段 | 持久，直到被推翻 | 语义 + 时间混合检索 |

生命周期是这样流转的：每一轮对话先进 Working；当窗口快满或一段话题结束，触发一次**沉淀（consolidation）**，把 Working 里的原始轮次压成 Short-term 摘要，同时抽取出值得长期保留的事实写进 Long-term。Working 会随窗口滑动自然丢弃，Short-term 按滚动窗口过期（比如只留最近 14 天的会话摘要），Long-term 只在被显式删除或被新事实推翻时才消失。"周二提到的地方"这条信息，理想路径是它在周二当天就被沉淀进了 Long-term，带着 `event_time=周二` 的时间戳，第三天才检索得到。

## 写入触发：什么信号把 Working 沉淀进 Long-term

写入是整个系统里最需要克制的环节——模型不能把每句闲聊都塞进长期库，否则检索时全是噪声。我用一个**分类器 + 规则**的组合来决定哪些内容值得写。触发信号主要有四类：

- **实体（entity）**：出现具体的人名、地名、时间、机构、金额等命名实体。"想去清迈"里的"清迈"就是一个地点实体。
- **偏好（preference）**：用户表达的稳定倾向，"我不吃辣""我喜欢靠窗的座位"。
- **事件（event）**：带时间锚点的计划或已发生的事，"下个月要搬家""上周离职了"。
- **显式指令**：用户明确说"记住""别忘了"，这类置信度直接给满。

每条写入 Long-term 的记录都带一组固定元数据，缺一不可：

~~~text
MemoryRecord {
  id, text,                       // 事实正文
  owner_id, tenant_id,            // 归属与隔离，防跨用户污染
  source: turn_id / tool_result,  // 来源，可回溯
  entity_type, entities[],        // 分类与抽取出的实体
  confidence: 0.0~1.0,            // 置信度，显式记住=1.0
  event_time,                     // 事实发生的时间（周二）
  created_at,                     // 写入时间
  importance: 0.0~1.0,           // 重要度，见下
  version, superseded_by          // 冲突版本管理
}
~~~

写入伪代码大致如此：

~~~python
def maybe_write(turn, working):
    signals = classify(turn)          # 实体/偏好/事件/显式
    if not signals:
        return
    facts = extract_facts(turn, working)   # 用小模型抽结构化事实
    for f in facts:
        if pii_detected(f) and not user_consented(f):
            continue                       # 敏感信息需授权
        f.confidence = score_confidence(signals, f)
        f.importance = score_importance(f) # 显式>偏好>事件>普通实体
        dup = search_similar(f, owner=turn.owner, thr=0.92)
        if dup:
            reconcile(dup, f)              # 冲突走版本管理，见后
        else:
            vector_store.upsert(f)
~~~

关键点：写入是**异步**的，不阻塞对话主链路；抽取用便宜的小模型；置信度不是拍脑袋，而是由信号类型决定——用户亲口说的 > 从行为推断的。

## 检索算法：语义、时间、重要度的混合打分

检索绝不能只做向量相似度 top-k，那样会把"最像"的旧事实排在"刚说过"的新事实前面。我的做法是先由一个轻量步骤把用户 query 解析成**检索意图**：抽出其中的实体、时间约束（"周二"→一个日期区间）、主题词，再用这个意图去向量库做混合打分。核心公式：

~~~text
score = α · sim(q, m) + β · recency(m) + γ · importance(m)

其中：
  sim        = query 与记忆向量的余弦相似度，∈[0,1]
  recency    = exp(-λ · Δt)，Δt 为 now 与 m.event_time 的间隔（天）
  importance = 写入时打的重要度分
  α, β, γ    典型取 0.6 / 0.25 / 0.15，且 α+β+γ=1
  λ          衰减常数，λ=ln(2)/half_life，半衰期决定"多久算旧"
~~~

`recency` 用指数衰减而不是线性，是因为记忆的时效价值前期掉得快、后期趋平——半衰期设 7 天时，一周前的事实新鲜度权重约 0.5，一个月前约 0.06。`half_life` 该按记忆类型分档：偏好类事实（"不吃辣"）半衰期设几个月甚至不衰减，事件类（"下周开会"）设几天。这样"稳定偏好"不会因为时间久就被压下去，而"临时安排"过期后自然沉底。

打分之外还有两道**硬过滤**先行：一是权限过滤（`owner_id`/`tenant_id` 必须匹配，跨用户记忆一律不召回）；二是当 query 带明确时间约束时，先按 `event_time` 落在区间内做**候选集裁剪**，再在候选集里算混合分。硬过滤保证正确性，混合打分负责排序。

## "周二提到的那个地方"如何命中

现在把上面的机制串到开头那个查询。用户问句先被解析成检索意图：`{时间约束: 上个周二 → [周二00:00, 周二23:59], 主题: 地点/想去的地方}`。检索流程：

1. **时间过滤**：在 Long-term 里筛出 `event_time` 落在周二区间的记录，候选集从上万条缩到几十条。
2. **语义召回**：用"想去的地方/目的地"生成 query 向量，在候选集内算 `sim`。周二那条"想去清迈"因为实体类型是地点、语义高度相关，`sim` 拉满。
3. **混合打分**：候选集内 `event_time` 都在同一天，`recency` 拉不开差距，于是 `sim` 和 `importance` 主导排序，"清迈"这条排到第一。
4. **回答**：把 top 记录连同 `source`（周二的原始 turn_id）一起交给模型，模型据此回答并能溯源。

命中的本质是：时间约束把搜索空间切小、保证"周二"这个维度不被语义相似度稀释，语义打分再在小空间里精准定位主题。任何一层单独都做不到——纯语义会把三天里所有聊过的地点混在一起，纯时间过滤又无法从周二几十条记录里挑出"地点"那条。

## 遗忘与冲突：让库保持干净

只写不删，库会越来越脏，检索精度随之下降。遗忘有两种：**被动衰减**——`recency` 权重让旧的临时事实在打分中自然沉底，形同软遗忘；**主动清理**——定时任务扫描并硬删三类记录：已过 `event_time` 且未被复用的临时事件、长期置信度低于阈值的事实、超过 N 天从未被检索命中的记忆。硬删要同步清理向量、缓存和评测样本，并留最小审计。

冲突事实用**版本链**而非覆盖。用户先说"想去清迈"，两天后改口"还是去大理吧"，两条都保留：新记录写入时把旧记录的 `superseded_by` 指向自己，旧记录降权但不删。检索默认只返回版本链的头部（最新有效版本），但当用户问"我之前是不是说过清迈"时，仍能沿链回溯出历史。这样既保证当前答案用最新事实，又不丢失"改过主意"这个信息本身。

## 边界与取舍

这套架构不是免费的。写入的分类器和事实抽取会引入额外的模型调用和延迟，虽然异步但增加了成本；打分公式里的 `α/β/γ/half_life` 需要用真实查询集调，没有普适最优值，调错了要么召回噪声要么漏掉旧事实。时间过滤依赖 `event_time` 抽取准确，用户说"周二"而系统解析错日期区间就会整体失准，相对时间（"上周""前几天"）的归一化尤其容易出错。实体消歧也有坑："那个地方"到底指哪个，多轮指代消解本身就不完美。我的经验是：把这套系统当成提升召回上限的工具，而不是精度保证，任何单条记忆返回时都带上来源和时间让用户可核对，比追求打分公式的完美更实在。
