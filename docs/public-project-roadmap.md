# 公开项目路线图

## 目标

个人博客只展示已经完成源码整理、安全检查和使用说明的真实项目。每个作品对应一个独立 GitHub 公共仓库，项目卡提供真实界面截图、实现说明和源码入口。

计划使用的 GitHub 组织方式：`https://github.com/xrlnewman/<repository>`。

## 公开门槛

项目从本地候选进入博客前必须同时满足：

1. 可从干净目录安装并运行，核心流程经过实际点测；
2. 提供 README、MIT License、环境要求、启动命令和构建命令；
3. 删除 Gitee 地址、内部地址、真实业务数据、客户信息和个人凭据；
4. 检查当前文件和完整 Git 历史，不包含 Token、密码、证书、数据库连接或私有接口；
5. 提供真实运行截图，不使用与最终产品不一致的效果图；
6. GitHub 仓库公开后再把项目设为 `draft: false`。

## 候选优先级

| 优先级 | 仓库 | 产品价值 | 当前判断 | 公开前工作 |
|---|---|---|---|---|
| P0 | `field-notes` | 项目作品、文章、搜索、评论和留言的一体化个人站点 | 首个公开作品 | 创建 GitHub 仓库、启用 Discussions、部署 Cloudflare Pages |
| P1 | `json-csv-sql` | JSON、CSV、Excel、SQL 等格式互转，适合开发和数据处理场景 | 体量小、边界清晰，适合作为第二个作品 | 补许可证、测试典型转换与异常输入、替换旧仓库地址、生成真实截图 |
| P1 | `image-batch` | 本地批量压缩、缩放、转格式和水印，用户价值直观 | 无服务端依赖，适合公开演示 | 补许可证、核对 README 与实际功能、测试大图和异常格式、生成截图 |
| P1 | `websocket-tester` | WebSocket 连接、收发日志、心跳和历史会话调试 | 功能集中，开发者受众明确 | 补许可证、检查自定义 Header 的本地存储、增加本地回显服务测试、生成截图 |
| P2 | `pdf-batch` | PDF 合并、拆分、提取、水印和转图片 | 使用场景广，但文件兼容性测试成本更高 | 补许可证、准备脱敏样例、测试加密/损坏/大文件、替换旧仓库地址 |
| P2 | `api-bench` | API 并发、QPS、分位延迟和错误分类 | 技术展示价值高，同时存在误用风险 | 人工复核敏感模式命中文件、增加安全使用说明和保守默认值、验证指标准确性 |
| P3 | `db-snapshot-diff` | MySQL/PostgreSQL 结构与采样数据对比 | 能体现复杂工程能力，但会接触数据库凭据和业务数据 | 审计凭据存储、默认脱敏、只读权限说明、迁移 SQL 安全测试、准备本地示例库 |
| 暂缓 | `toolkit-box` | 聚合多个本地效率工具 | 与独立工具重复，当前没有独立 Git 历史 | 先发布单项工具；聚合版稳定后再决定是否公开 |

## 初步审计记录

- `json-csv-sql`、`pdf-batch`、`image-batch`、`websocket-tester`、`api-bench`、`db-snapshot-diff` 当前工作区均无未提交改动；
- 上述仓库目前只有旧 Gitee 远端，公开 GitHub 时不直接复用远端配置；
- 初步文件名检查未发现被追踪的 `.env`、`.pem` 或 `.key`；
- 常见敏感模式的初筛仅在 `api-bench` 命中一个文件，需要人工查看上下文后才能判断是否安全；
- 所有候选都缺少实际的 `LICENSE` 文件，README 中出现 MIT 字样不能替代许可证文件；
- 初筛只用于排优先级，不能替代完整源码、依赖、构建产物和 Git 历史审计。

## 发布顺序

1. `field-notes`：先形成稳定的作品展示入口；
2. `json-csv-sql`：验证“一项目一仓库”的完整发布流程；
3. `image-batch`：补充面向普通用户的可视化工具；
4. `websocket-tester`：补充面向开发者的调试工具；
5. 其余项目根据安全审计和测试结果逐个进入博客。
