# 个人博客项目源码展柜扩充设计

## 背景与目标

当前博客只有一个公开项目和一个草稿项目，无法充分体现七年全栈开发经验。第一批从 `<workspace>` 中选择八个可公开项目，扩充为九个公开项目，并保证每张项目卡都有真实封面、详情说明和可访问的 GitHub 源码地址。

用户已明确授权自主决策，不再逐项确认。此次只公开个人演示项目，不公开公司游戏、支付、后台业务代码，也不公开本机凭据、日志、数据库和备份。

## 方案选择

采用“博客仓库内置源码展柜”的混合方案：源码快照放在 `showcase/<slug>/`，项目页的 `repoUrl` 指向 GitHub 对应子目录。后续可把最成熟的项目拆成独立仓库。

未采用的方案：

- 每个项目独立建仓：专业度高，但首次扩充需要维护八套仓库、许可证和发布流程；
- 只增加项目卡：无法满足源码必须可查看的要求。

## 信息架构与页面表现

- 首页继续显示项目优先区，优先展示四个最能形成能力组合的项目：个人博客、ToolkitBox、Invoice OCR、Inventory System。
- 项目列表保留五个固定分类和 URL 筛选；每个分类至少有一个项目。
- 项目卡展示真实封面、分类、状态、技术栈、项目详情和 GitHub 源码入口。
- 项目详情统一包含：项目目标、核心能力、技术实现、工程取舍、源码说明。
- 源码入口使用 `https://github.com/xrlnewman/field-notes/tree/main/showcase/<slug>`。

## 源码收录边界

只复制运行所需源码、README、依赖锁文件和示例配置；不复制原仓库 `.git` 元数据。

必须排除：

- `node_modules/`、`vendor/`、`dist/`、构建缓存；
- `.env`、真实密码、Token、私钥、Cookie；
- `memory.db`、备份目录、日志和本地索引；
- 公司业务数据、客户名称、生产地址；
- Gitee 远端信息。

## 封面生成

七个 Electron 项目通过本地静态服务加载其 renderer 页面，使用 1440×900 视口生成真实界面截图。若页面依赖 Electron preload，注入最小只读 mock 仅用于让初始界面稳定渲染，不伪造业务结果。

Inventory System 使用仓库内 SVG 架构封面，清楚标注为“系统架构”，不冒充产品运行界面。所有封面统一为 16:10，存放在 `public/images/projects/`。

## 代码与数据流

1. 从本地候选目录生成脱敏源码快照到 `showcase/`；
2. 安全扫描快照，发现凭据或运行时数据则阻断提交；
3. 生成项目封面并加入静态资源；
4. 为八个项目新增 Astro Content Markdown；
5. 构建时由现有项目集合自动生成首页卡片、分类计数和详情页；
6. 项目源码按钮跳转 GitHub 子目录。

## 错误处理与安全

- 任一项目未通过敏感信息扫描，不进入本批次；
- 任一源码链接没有对应 `showcase` 目录，测试失败；
- 任一封面不存在或尺寸不合格，测试失败；
- 不修改原项目的 Gitee 远端和历史，只读取本地文件；
- 不在公开 README 中声明未经验证的性能数字或客户案例。

## 测试与验收

- 内容测试：项目内容总数为十、公开项目数为九，八个新增 slug、分类、技术栈和源码链接正确；
- 文件测试：每个新增项目的 `showcase/<slug>/README.md` 与封面存在；
- 安全测试：源码快照不包含禁止文件名和常见密钥格式；
- 构建测试：24 个静态页面可生成，首页和项目列表包含新增项目；
- 浏览器点测：桌面与手机视口检查项目卡、分类筛选、详情页和源码链接；
- 线上验收：首页、项目列表、八个详情页和 GitHub 子目录均返回成功。

## 非目标

- 本批次不创建八个独立 GitHub 仓库；
- 不为桌面工具提供安装包下载；
- 不公开公司项目；
- 不把演示项目包装成已有商业客户或虚构收入。
