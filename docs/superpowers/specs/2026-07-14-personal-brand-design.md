# 许汝林个人博客品牌文案设计

## 目标

把现有通用博客身份升级为“许汝林个人博客”，核心定位是产品型全栈工程师，辅助气质是持续构建、公开记录的独立创造者。站点以可运行、可查看源码的项目作品为主，技术文章为辅。访问者应迅速理解四件事：作者是谁、做过什么项目、具备什么能力、能解决什么问题。

## 定位与边界

- 姓名：许汝林。
- 当前信息：27 岁，7 年开发经验。
- 核心角色：产品型全栈工程师。
- 公开业务表述统一使用“交易系统、运营平台、数据系统与复杂后台”，不披露具体行业、客户或公司。
- 可以包装能力和表达方式，但不编造公司、职级、收入、用户量、性能指标等需要证据支撑的数据。
- GitHub 使用已连接账号 `xrlnewman`，只链接已经真实创建并包含源码的公开仓库。
- 不公开个人邮箱，也不展示空白或虚假的其他社交链接。

## 文案方向

采用“产品工程师为主、独立创造者为辅”的组合：

- 产品工程师负责建立专业信任，强调从需求到上线的完整交付能力。
- 独立创造者负责形成个人感，解释为什么持续开发产品、写文章和公开复盘。
- 简历式技术清单放在关于页，不让首页首屏变成关键词堆砌。

## 站点身份

- 站点名：`许汝林个人博客`
- 品牌字标：`许汝林 / PRODUCT ENGINEER`
- 品牌方块：`许`
- 首页标题：`许汝林个人博客｜产品型全栈工程师`
- 默认描述：`许汝林，27 岁，拥有 7 年开发经验的产品型全栈工程师。专注把复杂业务转化为稳定、易用、可持续迭代的产品。`
- 作者简介：`我是一名拥有 7 年开发经验的产品型全栈工程师，能从需求梳理、系统设计、前后端开发一路做到部署交付与持续迭代。`
- GitHub：`https://github.com/xrlnewman`

年龄和经验年限作为结构化配置保存，方便以后统一更新。

## 首页

首屏保持现有 Editorial Green 视觉语言，但压缩纵向占用，让项目尽快进入首屏下方：

- 眉题：`PRODUCT ENGINEER · BUILDER · WRITER`
- 主标题：`把复杂业务，做成真正可用的产品。`
- 简介使用站点默认描述。
- 主按钮仍为“查看项目”，次按钮仍为“阅读文章”。
- 状态标签：`7 年持续交付`
- 当前关注标题保持不变。
- 当前关注说明：`把工程经验转化为可复用的产品、工具和文章，持续记录从需求判断到上线交付的完整过程。`

首屏之后立即进入“项目作品”区域；文章区域后置并降低视觉权重。

## 项目优先的信息架构

参考用户提供的项目商城截图，吸收“分类筛选 + 截图卡片 + 密集陈列”的信息结构，不复制商城的销量、评分、退款和价格信息。

首页顺序调整为：

1. 紧凑个人价值首屏。
2. 精选项目：默认展示 6 个以内项目，提供“查看全部项目”。
3. 最新文章：作为开发过程和技术判断的补充证据。

项目索引页包含：

- 顶部项目总览与分类筛选。
- 分类使用易理解的作品类型，例如“全部、开发工具、数据工具、AI 应用、网站产品、业务系统”。
- 三列桌面项目网格；中等屏幕两列；移动端一列。
- 筛选在浏览器本地完成，不请求后端；按钮可键盘操作并暴露选中状态。
- 无匹配结果时显示明确空状态。

## 项目卡片与详情

项目卡片使用真实产品截图作为主要视觉，信息顺序为：

1. 项目封面或界面截图。
2. 项目分类与当前状态。
3. 项目名称和一句话价值。
4. 最多三个核心技术标签。
5. “查看项目”和“GitHub 源码”两个明确入口。

整张卡片进入站内项目详情；GitHub 按钮直接打开对应公开仓库，二者不能互相遮挡或形成嵌套链接。

项目详情页保留现有长文结构，并增加大图封面。页面头部持续提供“在线体验”和“GitHub 源码”；正文说明问题、方案、关键实现、运行方式与取舍。

## 项目内容模型

公开项目至少包含：

- `title`、`description`、`publishedAt`、`status`、`tech`。
- `category`：用于项目筛选。
- `cover`：仓库内的真实截图或产品封面。
- `repoUrl`：真实存在的 GitHub 公开仓库地址。
- `demoUrl`：有在线版本时提供，没有则不显示。

非草稿项目必须提供 `cover` 和 `repoUrl`，构建期校验缺失字段，禁止产生没有源码或使用占位链接的公开项目卡片。

## GitHub 源码策略

- 采用“一项目一仓库”，统一放在 `https://github.com/xrlnewman/` 下。
- 博客仓库只保存项目介绍、截图和入口，不复制其他项目源码，避免形成难维护的大型混合仓库。
- 一个项目只有在仓库包含可运行源码、README、启动说明和许可证后才把 `draft` 改为 `false`。
- 现有本地工具按“安全检查 → 清理密钥与私有数据 → 补 README/License → 本地验证 → 建公开仓库 → 推送 → 添加博客项目”的顺序逐个公开。
- 第一批优先整理已经有完整源码、可独立运行且容易说明价值的工具；未完成安全审计的业务仓库不得公开。

## SEO 与可访问性

- 首页 title 与 description 使用新的个人身份信息。
- 关于页 title 使用“关于许汝林”，description 概括 7 年经验与能力边界。
- 页头链接的可访问名称跟随新站点名。
- RSS、Open Graph、页脚版权和站点地图中的站点名称通过统一配置自动同步。
- 品牌改动不得破坏深浅主题、移动端导航和现有键盘操作。

## 验收标准

1. 页面中不再出现 `Field Notes`、作者占位符 `Y` 或“独立开发者”旧身份。
2. 首页首屏完整呈现姓名、7 年经验、产品型全栈定位和解决问题的价值表达。
3. 关于页呈现年龄、经验、完整交付能力、复杂业务能力和核心技术栈。
4. 首页项目优先于文章，项目索引支持无后端分类筛选。
5. 所有公开项目都有真实封面和有效 GitHub 地址；项目卡可进入详情，也可直接打开源码。
6. 邮箱和空社交链接不渲染，只公开已连接的 GitHub 主页。
7. 390px 移动端无横向滚动，品牌名、数据栏、筛选器和项目卡均可读。
8. 单元测试、Astro 检查、生产构建和构建产物测试全部通过。
