# 十个高需求产品扩展设计

## 目标

将许汝林个人博客从现有 10 个项目扩展到 20 个项目。新增项目必须是可启动、可操作、可复盘的产品闭环，不使用只有静态按钮的空壳页面。

## 新增产品矩阵

| 产品 | 业务闭环 | 分类 |
| --- | --- | --- |
| HireFlow 招聘协同 | 发布职位 → 收集候选人 → 面试 → 录用/淘汰 | 人力资源 |
| FeeFlow 企业费控 | 申请 → 审批 → 报销 → 付款/归档 | 企业服务 |
| CRMFlow 销售协同 | 线索 → 跟进 → 商机 → 赢单/丢单 | 企业服务 |
| ContractFlow 合同管理 | 起草 → 审批 → 签署 → 履约提醒 → 归档 | 企业服务 |
| PropertyFlow 物业工单 | 报修 → 派单 → 上门 → 验收 → 评价 | 物业服务 |
| HelpdeskFlow 客服工单 | 创建工单 → 分派 → 回复 → 解决 → 满意度 | 客户服务 |
| EventFlow 活动票务 | 创建活动 → 售票 → 核销 → 现场服务 → 结算 | 文体活动 |

## 仓库和运行形态

每个产品拆为两个公开仓库：`<product>-admin` 包含 Go Gin API、运营后台、MySQL 8.4 schema/seed、Redis 8 幂等与锁、Docker Compose；`<product>-miniapp` 包含移动端/H5 体验和同一 `/api/v1` 客户端。所有写接口要求 `Idempotency-Key`，状态变化写入事件表，统一响应包含 `traceId`。

每个后台支持两种模式：未配置外部依赖时使用合成内存数据用于 UI 预览；配置 `MYSQL_DSN`、`REDIS_ADDR` 后使用 MySQL/Redis 真实闭环。Compose 必须定义 MySQL 8.4、Redis 8 健康检查并等待依赖健康后启动 API。

## UI 和截图验收

每个后台至少提供：总览、主业务列表、状态/异常处理、结算或分析页；移动端至少提供：首屏 KPI、主流程操作、状态卡片、提醒。页面使用不同的行业色板，不把所有项目都做成绿色；数据使用 8~20 条虚构记录，列表字段、空状态、错误状态和完成状态都可见。截图必须在构建后通过浏览器检查，避免文字溢出、图片变形和移动端截断。

## 博客集成

为每个产品创建一篇项目内容，包含分类、封面、至少 4 张截图、两个 GitHub 仓库关联、技术栈、完整闭环说明和免费开源说明。项目页支持按行业分类筛选，详情页展示两仓库链接；博客只声明展示站已上线，不虚报后端公网地址，API 通过 Compose 可复现。

## 验证标准

- 每个后台：`go test ./...`、`go vet ./...`、`docker compose config`、Docker 镜像构建通过。
- 每个前端：`npm test`、`npm run build`、`npm audit --omit=dev --audit-level=moderate` 通过。
- 每个 API 至少验证一条创建/状态推进/完成流程和同一幂等键重复请求只落一次。
- 博客：内容测试、Astro check、构建和截图引用检查通过；线上新项目 URL 返回 HTTP 200。
