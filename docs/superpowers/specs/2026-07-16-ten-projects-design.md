# 十个高需求项目矩阵设计

## 目标

将个人博客从 8 个公开项目扩展到 10 个，新增两个可运行感强、截图排版稳定、源码关联完整的产品，并对现有项目的截图、色彩、数据密度和展示页做一次全量回归。

## 现有项目基线

博客现有 8 个项目：Field Notes、HomeFlow、邻里社区、多商户商城、SkyBoom 企业官网、StoreFlow、StockFlow、EduFlow。已有项目原则上不改业务逻辑，只修复确认存在的展示问题、截图问题和元数据问题。

## 新增产品

### 9. FleetFlow 同城运输与配送管理平台

面向 3–50 辆车的同城配送车队、批发商配送中心和第三方物流团队。核心闭环为：客户下单 → 运单池 → 按区域/时效派车 → 司机接单 → 装货/到店/签收节点 → 异常上报 → 运费结算与对账。它只负责运输履约，不承担 StockFlow 的 SKU 库存管理。

公开代码拆分为两个仓库：

- `fleetflow-miniapp`：司机/收货端，包含待接单、运输中、扫码签收、异常上报和历史运单。
- `fleetflow-admin`：调度后台和 Go Gin API，包含运单池、调度看板、车辆、司机、异常、对账和数据看板。

演示数据必须看起来像正在运行：今日运单 128、待派车 24、运输中 67、准时率 96.8%、异常件 5、待对账 ¥86,420；提供 7 日运单趋势、区域分布、车辆利用率和异常类型图表。

视觉采用深海军蓝 `#0F1B33`、电光蓝 `#3B82F6`、青绿 `#19C3A6`，橙色 `#FF9D4D` 仅用于告警和主动作；地图/路线/节点时间线是核心视觉，不使用纯绿色单色界面。

### 10. CareFlow 诊所预约与健康随访平台

面向口腔、体检、中医和专科诊所。核心闭环为：选择门店/科室 → 查看医生排班 → 分时预约 → 到店签到/候诊 → 服务记录 → 复诊提醒 → 随访问卷 → 满意度回收。产品明确不包含诊断、处方、真实医疗数据和支付结算。

公开代码拆分为两个仓库：

- `careflow-miniapp`：患者端，包含门店/科室选择、医生排班、预约、候诊状态、就诊记录和随访问卷。
- `careflow-admin`：诊所运营后台和 Go Gin API，包含运营总览、预约日历/候诊队列、医生排班、患者档案、随访任务和满意度分析。

演示数据必须看起来像正在运行：今日预约 86、已签到 54、待随访 31、爽约率 4.6%、本周复诊率 38.2%、满意度 4.8/5；提供预约时段热力、科室转化、随访完成率和复诊趋势。

视觉采用暖白 `#F8FAFC`、靛蓝 `#4F46E5`、薄荷青 `#10B981`、珊瑚 `#F97316`，医疗蓝灰用于次要信息，卡片统一 16px 圆角、清晰状态色和高可读行高。

## 统一技术约束

- 前端：Vue 3/TypeScript 或 uni-app Vue 3；学员/司机/患者端必须能构建 H5 演示。
- 后台：Vue 3 + Vite 响应式运营后台。
- 服务端：Go 1.25 + Gin，统一 `{ code, message, data, traceId }` 响应信封。
- 数据：MySQL 8.4、Redis 8、Docker Compose；Redis 用于幂等键、派单锁或号源锁。
- 所有写接口支持幂等键，列表接口强制分页；演示模式不依赖外部服务也能展示完整数据。
- README 必须声明免费、开源、可自部署，并说明演示数据为虚构数据。

## 截图与视觉验收规范

- 桌面后台统一使用 1045×1224 正常桌面视口截图；禁止以移动视口或缩放后的视口截图冒充桌面端。
- 移动端统一使用 390×844 设备视口截图；必须保留设备边框、听筒和底部指示条。
- 每个新项目至少提供 4 张截图：总览、核心列表/日历、业务详情或任务、移动端关键流程。
- 每张截图必须有足够真实数据，不使用空状态、单行列表或大面积无意义留白作为主图。
- 逐张检查文字是否横排、表格是否溢出、按钮是否被截断、卡片是否重叠、状态色是否与语义一致；检查通过后才写入博客 markdown。
- 所有截图的 markdown `width`/`height` 必须与真实文件一致，图片优先控制在 1MB 内。
