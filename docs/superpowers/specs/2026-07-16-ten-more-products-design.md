# 再增十个高需求产品设计

## 目标

将个人博客从 20 个项目扩展到 30 个项目。新增项目必须是可启动、可操作、可复盘的产品闭环，不做只有静态按钮的展示壳；每个项目的后台/API、移动端、数据、截图和源码链接必须互相对应。

## 新增产品矩阵

| 产品 | 业务闭环 | 分类 |
| --- | --- | --- |
| InvoiceFlow 企业开票与收款 | 开票申请 → 审核 → 开票 → 收款核销 → 对账归档 | 财务管理 |
| RepairFlow 售后维修协同 | 报修 → 诊断 → 报价 → 维修 → 验收 → 回访 | 售后服务 |
| SupplyFlow 采购供应链 | 采购申请 → 询价 → 比价审批 → 到货验收 → 结算 | 采购供应链 |
| PayrollFlow 薪酬绩效 | 员工入职 → 绩效目标 → 薪资核算 → 审批发放 → 月度归档 | 人力资源 |
| BookingFlow 预约排班 | 服务配置 → 预约 → 排班 → 到店核销 → 评价复购 | 生活服务 |
| LabFlow 医疗样本 | 送检登记 → 样本接收 → 检验 → 复核 → 报告归档 | 医疗健康 |
| TravelFlow 旅行酒店 | 产品上架 → 询价预订 → 支付确认 → 出行服务 → 售后结算 | 旅游住宿 |
| CreatorFlow 内容排期 | 选题 → 脚本 → 制作 → 审核 → 发布 → 数据复盘 | 内容创作 |
| LegalFlow 法务案件 | 委托登记 → 立案 → 任务协同 → 节点提醒 → 结案归档 | 法律服务 |
| VenueFlow 场馆运营 | 场地配置 → 预约售票 → 入场核销 → 活动执行 → 日结分析 | 场馆运营 |

## 仓库和运行形态

每个产品拆为两个公开仓库：`<product>-admin` 包含 Go Gin API、运营后台、MySQL 8.4 schema/seed、Redis 8 幂等与锁、Docker Compose；`<product>-miniapp` 包含移动端/H5 体验和同一 `/api/v1` 客户端。所有写接口使用 `Idempotency-Key`，状态变化写事件记录，响应保持 `{ code, message, data, traceId }`。

后台在未配置外部依赖时使用合成演示数据，配置 `MYSQL_DSN`、`REDIS_ADDR` 后执行真实闭环。Compose 定义 MySQL 8.4、Redis 8 健康检查并等待依赖健康后启动 API。所有数据为虚构演示数据，不接入真实医疗、支付或身份信息。

## UI 和截图验收

每个后台至少呈现总览、主业务列表、异常/待处理、结算或分析四类视图；移动端呈现首屏 KPI、主流程操作、状态卡片和提醒。十个产品使用不同的行业色板，数据保持 8—20 条可读记录，避免统一绿色、溢出和图片变形。每个项目构建后通过浏览器检查，保存 4 张桌面/移动端截图。

## 博客集成

每个项目新增一篇内容，包含分类、封面、4 张截图、两个 GitHub 仓库、技术栈、完整闭环和免费开源说明。项目列表扩展至 30 个，分类筛选统计与详情页链接同步更新；博客只声明展示站公网可访问，后端通过 Compose 可复现，不虚报公网 API。

## 验证标准

- 每个后台通过 `go test ./...`、`go vet ./...`、`docker compose config`。
- 每个前端通过 `npm test` 与对应构建命令。
- 每个 API 冒烟验证创建、状态推进、完成流程及重复幂等键只落一次。
- 博客通过内容测试、Astro check、构建和截图资源检查，线上新增详情页返回 HTTP 200。
